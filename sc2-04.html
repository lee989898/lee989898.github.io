Set-UID Privileged Programs

Set-UID Privileged Programs

권한 있는 프로그램의 필요성

  • 비밀번호 딜레마
    • /etc/shadow 파일의 권한:
    • 일반 사용자는 비밀번호를 어떻게 변경할까?

Two-Tier 접근

  • 세분화된 액세스 구현 운영 체제의 제어가 OS를 만든다 -> 너무 복잡하다
  • OS는 extension에 의존하여 세분화된 액세스 제어한다
  • 권한 있는 프로그램은 그러한 extension이다

Privileged 프로그램의 종류

  • Daemons
    • 백그라운드에서 실행되는 컴퓨터 프로그램
    • 루트 또는 기타 권한이 있는 사용자로 실행해야 한다
  • Set-UID Programs
    • UNIX 시스템에서 널리 사용
    • 특수 비트로 표시된 프로그램

슈퍼맨 이야기

  • Power Suit
    • 슈퍼피플: 그들에게 직접 힘을 준다
    • 이슈: 나쁜 슈퍼피플
  • Power Suit 2.0
    • 컴퓨터 칩
    • 특정 작업
    • 사전 프로그래밍된 작업에서 벗어날 방법 없음
  • Set-UID 매커니즘 : 파워 슈트 메커니즘 리눅스 OS에서 구현

Set-UID Concept

  • 사용자가 프로그램 소유자의 권한으로 프로그램을 실행할 수 있도록 허용한다
  • 사용자가 일시적으로 상승된 권한으로 프로그램을 실행할 수 있도록 허용
  • Example: the passwd program
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root 41284 Sep 12 2012 /usr/bin/passwd
  • 모든 프로세스에는 두 개의 사용자 ID가 있다
  • Real UID (RUID): 프로세스의 실제 소유자 식별
  • Effective UID (EUID): 프로세스의 권한을 식별
    • 액세스 제어는 EUD를 기반으로 한다
  • 일반 프로그램이 실행될 때 RUID = EUID,프로그램을 실행하는 사용자의 ID로 둘 다 동일하다
  • Set-UID가 실행되면 RUID ≠ EUID. RUID는 여전히 같다. 사용자 ID이지만 EUID는 프로그램 소유자 ID와 동일하다
    • 프로그램이 root에 의해 소유된 경우 프로그램은 root 권한으로 실행된다

프로그램을 Set-UID로 전환

  • 소유자 변경 -> root에 대한 파일로
  • 활성화하기 전에 Set-UID 비트
  • 활성화 후 Set-UID 비트

작동 원리

Set-UID 프로그램은 Set-UID 비트라고 하는 단일 비트인 특수 마킹을 제외하고는 다른 프로그램과 같다

UID 설정 예

privileged program 아님

privileged program 임

privileged program 이지만 root privilege는 아님

Set-UID는 어떻게 안전한가?

  • 일반 사용자가 권한을 에스컬레이션할 수 있다
    • 권한을 직접 부여하는 것과는 다르다(sudo 명령)
    • 제한된 행동 – 슈퍼맨이 디자인한 컴퓨터 칩과 유사
  • 모든 프로그램을 Set-UID로 전환하는 것은 안전하지 않다
    • 예: /bin/sh
    • 예: vi

슈퍼맨에 대한 공격

  • 사용자가 코딩된 작업만 수행할 수 있다고 가정할 수 없음
    • 개발자의 코딩 결함
  • 슈퍼맨 말로이
    • 북쪽으로 비행한 다음 좌회전
    • 이 코드를 어떻게 악용하는가?
  • 슈퍼맨 말로리
    • 북쪽으로 날아 서쪽으로 회전
    • 이 코드를 어떻게 악용하는가?

Set-UID 프로그램의 표면 공격

사용자 Input을 통한 공격

사용자 Input: 명시적 입력

  • 버퍼 오버 플로우
    • 악성코드 실행을 위한 버퍼 오버플로우
  • 형식 문자열 취약점
    • 사용자 입력을 형식 문자열로 사용하여 프로그램 동작 변경
  • CHSH – 쉘 변경
    • 기본 셸 프로그램을 변경할 수 있는 Set-UID 프로그램
    • 쉘 프로그램은 /etc/passwd 파일에 저장된다
  • 문제
    • 사용자 입력 삭제 실패
    • 공격자가 새 루트 계정을 만들 수 있음
  • 공격

시스템 Input을 통한 공격

  • 경쟁 조건
    • 권한이 없는 파일에서 권한 있는 파일에 대한 심볼릭 링크
    • 인플루언서 프로그램
    • world writable 폴더 내부에 쓰기

환경변수를 통한 공격

  • 동작은 내부에 표시되지 않는 입력의 영향을 받을 수 있다

  • 환경 변수 : 사용자가 실행하기 전에 설정할 수 있다

  • PATH 환경 변수
    • 사용자가 명령의 전체 경로 제공하지 않은 경우 쉘 프로그램에서 명령을 찾는 데 사용한다
    • system(): /bin/sh를 먼저 호출한다
    • system(“ls”)
      • /bin/sh는 PATH 환경 변수를 사용하여 “ls”를 찾는다
      • 공격자는 PATH 변수를 조작하고 “ls” 명령을 찾는 방법을 제어할 수 있다

Capability Leaking

  • 몇몇 경우에 권한 있는 프로그램은 실행중에 스스로 다운그레이드 한다
  • 예: su 프로그램
    • 권한이 있는 Set-UID 프로그램이다
    • 한 사용자가 다른 사용자로 전환할 수 있다(예: user1에서 user2로)
    • 프로그램은 루트로 EUID를, user1로 RUID로 시작한다
    • 비밀번호 확인 후, EUID와 RUID는 모두 user2가 됩니다(권한을 통해 다운그레이드)
  • 이러한 프로그램은 기능 누출로 이어질 수 있다
    • 프로그램은 다운그레이드하기 전에 권한 있는 기능을 정리하지 못할 수 있다

Capability Leaking을 통한 공격: 예

sudo chown root ./prog sudo chmod 4755 ./prog

운영체제에서 실행 중인 프로세스에대해서 접근제어를 적용할대는 EUID를 기준으로 이 프로세스가 현쟈 하고자 하는 오퍼레이션을 허용할지 말지를 결정한다
EUID를 기준으로 시스템의 접근제어가 이루어진다

set-uid를 통해 공격이 들어오기 때문에 Set-UID 비트를 해제한다. 공격의 예로는 시스템 인풋, 유저 인풋, 환경 변수가 있다
