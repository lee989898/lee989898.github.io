환경변수와 공격들

Environment Variables

  • 동적으로 이름지어진 값의 집합
  • 프로세스가 진행되는 운영 환경의 일부
  • 실행 중인 프로세스가 동작하는 방식에 영향
  • Unix에 도입되었으며 Microsoft Windows에서도 채택됨
  • ex) PATH 변수
    • 프로그램이 실행될 때 쉘 프로세스는 환경을 사용한다. 전체 경로가 제공되지 않은 경우 변수를 사용하여 프로그램이 있는 위치를 찾는다

프로세스는 어떻게 환경 변수를 얻을까?

프로세스는 두가지 방법으로 환경 변수를 가져온다

  • fokr() 시스템 호출을 사용하여 새 프로세스가 생성되면 자식 프로세스는 부모 프로세스의 환경 변수를 상속한다

  • 프로세스가 자체적으로 새 프로그램을 실행하는 경우 일반적으로 execve 시스템 호출을 사용한다. 이경우 메모리 공간을 덮어쓰고 모든 이전 환경 변수가 사라진다. execve()는 전달하기 위해 특별한 방식으로 호출 될 수 있다

프로그램은 새로운 프로그램 /usr/bin/env, 현재 프로세스의 환경 변수를 출력한다
새로운 변수 newenv를 3번째로 사용

환경 변수의 메모리 위치

  • envp 및 환경은 처음에는 같은 장소를 가리킨다
  • envp는 내부에서 메인 함수에서 액세스할 수 있다. Environ은 전역 변수
  • 환경 변수에 변경 사항이 있을 때(예: 새로 추가됨), 환경변수의 저장 위치는 힙으로 바뀔 수 있다, 그래서 environ은 변경될 것이다(envp는 바뀌지 않는다)

셸 변수 및 환경 변수

  • 사람들은 종종 쉘 변수와 환경 변수를 똑같다고 착각한다
  • 쉘 변수
    • 쉘에서 사용하는 내부 변수
    • 쉘은 사용자가 쉘을 생성, 할당 및 삭제할 수 있도록 내장 명령을 제공한다
    • 예제에서는 FOO라는 쉘 변수를 생성한다
  • 쉘 프로그램이 시작될 때 환경 변수를 자체 쉘에 복사합니다. 자체 쉘 변수에 대한 변경 사항은 반영되지 않는다, 아래 그림은 그 예이다

  • 아래 사진은 쉘 변수가 자식 프로세스의 환경 변수에 영향을 미치는 방법을 보여준다
  • 또한 부모 쉘의 환경 변수가 자식 프로세스의 환경 변수가 되는 것을 보여준다

쉘 프롬프트에 env를 입력하면 쉘은 자식 프로세스를 생성한다

/proc 파일 시스템에 대한 참고 사항

  • /proc은 Linux의 가상 파일 시스템이다.프로세스 ID를 디렉터리 이름으로 사용 하여 디렉토리를 포함한다
  • 각 프로세스 디렉토리에는 Environ이라는 가상 파일이 포함되어 있다. 프로세스의 환경
    • 예를 들어 가상 파일 /proc/932/environ은 다음과 같은 환경 변수를 포함한다. 프로세스 932
    • “strings /proc/\(/environ" 명령은 현재 프로세스의 환경변수를 출력한다. (셸은\)를 자체 프로세스 ID로 대체함)
  • env 프로그램은 bash 셸에서 호출될 때 자식 프로세스에서 실행된다. 따라서 자기 것이 아닌 쉘의 자식 환경 변수를 출력한다

환경 변수에 대한 공격

  • 환경변수를 숨은 의도로 사용하면 매우 위험하다
  • 사용자가 환경 변수를 설정할 수 있기 때문에 그것들은 Set-UId 프로그램 공격의 일부가 된다

Dynamic Linker를 통한 공격

  • Linking은 프로그램에서 참조하는 외부 라이브러리 코드를 찾는 것이다
  • 링크는 런타임 또는 컴파일 시간 동안 수행할 수 있다
    • Dynamic Linking: 환경 변수를 사용하며, 공격 surface의 일부가 된다
    • Static Linking
  • 다음 예를 사용하여 정적 및 동적을 구별한다. 연결:

  • Static Linking
    • 링커는 프로그램 코드와 라이브러리 코드를 결합한다. printf() 함수를 포함하여
    • 정적 컴파일된 프로그램의 크기가 100배임을 알 수 있다. 동적 프로그램보다 크다

  • Dynamic Linking
    • Linking은 런타임 중에 수행된다
      • 공유 라이브러리(Windows의 DLL)
    • 동적 연결로 컴파일된 프로그램이 실행되기 전에 해당 실행 파일이 메모리에 먼저 로드됨

  • “ldd” 명령을 사용하여 어떤 공유 라이브러리가 의존하는 지 확인할 수 있다

Dynamic Linker를 통한 공격: the Risk

  • Dynamic Linkeking은 메모리를 저장한다. 프로그램 코드의 일부가 컴파일 시간동안 결정되지 않았음을 의미한다
  • 사용자가 누락된 코드에 영향을 줄 수 있는 경우 프로그램의 무결성을 compromise 할 수 있다

Review: Dynamic Linker

  • Linking은 프로그램에서 참조된 외부 라이브러리 코드를 찾는 과정을 말한다
  • Linking은 런타임 또는 컴파일 시간 동안 수행할 수 있다
    • Dynamic Linking – 환경 변수를 사용 (공격 표면)
    • 정적 연결

Dynamic Linker를 통한 공격 :Case Study 1

  • LD_PRELOAD는 linker가 먼저 검색할 공유 라이브러리 목록을 포함한다
  • 모든 기능을 찾지 못한 경우 링커는 여러 목록(LD_LIBRARY_PATH에 의해 지정된 폴더를 포함하는 폴더) 중에서 검색한다
  • 두 변수 모두 사용자가 설정할 수 있으므로 연결 프로세스의 결과 제어를 보여준다
  • 해당 프로그램이 Set-UID 프로그램인 경우 보안 문제로 이어질 수 있다

ex 1 - 일반적인 프로그램

프로그램은 동적으로 연결된 sleep 함수를 호출한다

#include <unistd.h>
int main()
{
  sleep(1);
  return 0;
}

->

이제 우리 소유의 sleep() 함수를 구현한다

#include <stiod.h>
void sleep(int s)
{
  printf("I am not sleeping!\n");
}

위의 코드를 컴파일하고 공유 라이브러리를 만들고 LD_PRELOAD 환경 변수에 공유 라이브러리를 추가해야 한다

ex 2 - Set-UID 프로그램

예제 1의 기술이 Set-UID 프로그램에서 작동하면 매우 위험하다 위의 프로그램을 Set-UID로 변환하자

  • sleep() 함수가 호출되지 않는다
    • 동적 링커에서 구현한 대책 때문입니다. EUID와 RUID가 다른 경우 LD_PRELOAD 및 LD_LIBRARY_PATH 환경 변수를 무시한다

Dynamic Linker를 통한 공격 :Case Study 2

Case: OS X Dynmaic Linker

  • 애플 OS X 10.10은 보안 영향을 완벽하게 분석하지 않고 새로운 환경 변수를 추가 했다
  • DYLD_PRINT_TO_FILE
    • 사용자가 dyld의 파일 이름을 제공할 수 있는 기능
    • Set-UID 프로그램인 경우 사용자는 보호된 파일에 쓸 수 있다
    • 기능 누출 – file descriptor가 닫히지 않음
  • 악용 예:
    • Set DYLD_PRINT_TO_FILE to /etc/sudoers
    • Switch to Bob’s account
    • The echo command writes to /etc/sudoers

외부 프로그램을 통한 공격

  • 응용 프로그램은 외부 프로그램을 호출할 수 있다
  • 응용 프로그램 자체는 환경 변수를 사용하지 않을 수 있지만 호출된 외부 프로그램은 사용할 수 있다
  • 외부 프로그램을 호출하는 일반적인 방법:
    • execve()를 호출하는 함수의 exec() 계열: 프로그램을 직접 실행
    • system()
      • system() 함수는 execl()을 호출한다
      • execl()은 결국 execve()를 호출하여 /bin/sh를 실행한다
      • 쉘 프로그램은 프로그램을 실행한다
  • 이 두 접근 방식은 공격 표면이 다르다

외부 프로그램을 통한 공격: Attack Surfaces

  • system()에 비해 execve()의 공격면이 적음
  • execve()는 셸을 호출하지 않으므로 환경변수의 영향을 받지 않는다
  • 권한 있는 프로그램에서 외부 프로그램을 호출할 때 execve() 사용

라이브러리를 통한 공격

프로그램은 종종 외부 라이브러리의 기능을 사용한다. 이러한 기능의 경우 환경 변수를 사용하여 공격 표면에 추가 ex – UNIX의 Locale

  • 메시지를 인쇄해야 할 때마다 프로그램은 번역된 메시지에 대한 라이브러리 기능을 사용한다
  • Unix는 libc 라이브러리에서 gettext() 및 catopen()을 사용한다
  • 다음 코드는 프로그램이 로케일 하위 시스템을 사용하는 방법을 보여준다

  • 이 하위 시스템은 다음 환경 변수에 의존한다. LANG,LANGUAGE, NLSPATH, LOCPATH, LC_ALL, LC_MESSAGES
  • 이러한 변수는 사용자가 설정할 수 있으므로 번역된 메시지는 사용자에 의해 제어될 수 있다
  • 공격자는 형식 문자열 취약점을 사용하여 printf() 형식을 지정할 수 있다

어플리케이션 코드를 통한 공격

  • 프로그램은 환경 변수를 직접 사용한다. 만약 privileged 프로그램이면, 신뢰할 수 없는 입력이다

  • 프로그램은 getenv()를 사용하여 PWD의 현재 디렉토리 안다
  • 프로그램은 이것을 배열에 복사합니다. 길이 확인하는 것을 잊으면 잠재적인 버퍼 오버플로가 있다
  • PWD의 값은 셸에서 가져온다. 프로그램을 변경할 때마다 폴더 쉘 프로그램 업데이트
  • 내가 쉘 변수를 변경할 수 있다
